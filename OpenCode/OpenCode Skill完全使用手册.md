## 一、Skill 核心概念

### 1.1 定义与本质

Skill 是**带 YAML 元数据的 Markdown 文件**，存储在特定目录中，包含：
- **元数据 (Frontmatter)**：技能名称、描述、许可证等
- **内容主体**：功能说明、使用场景、步骤、示例等

**核心价值**：将专业知识与工作流程"封装"，实现**一次编写、多次复用**，解决 AI 输出不一致、流程不规范等问题。

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **按需加载** | 代理根据任务自动选择或用户手动调用 |
| **权限控制** | 支持全局/代理级权限，可`allow`/`deny`/`ask` |
| **多级发现** | 项目级→全局级→自定义路径，优先级递减 |
| **无参数设计** | 当前版本技能无参数，提供静态指令 (未来将支持) |
| **跨平台兼容** | 支持 Claude Desktop 技能格式 |

## 二、内置 Skill 清单

### 2.1 OpenCode Native 核心技能

OpenCode Native 提供以下核心内置技能，覆盖开发全流程：

| 技能名称 | 核心功能 | 适用场景 |
|----------|----------|----------|
| **`/batch`** | 大规模代码变更编排 | 批量重构、跨文件修改、项目迁移 |
| **`/review`** | 代码审查与质量检查 | PR 审核、代码规范验证、安全漏洞扫描 |
| **`/commit`** | 标准化提交与消息生成 | 遵循 Conventional Commits 规范提交代码 |
| **`/explore`** | 代码库探索与分析 | 新项目熟悉、代码结构理解、依赖分析 |
| **`/websearch`** | 网络搜索集成 | 技术调研、API 文档查询、解决方案查找 |
| **`/git-master`** | Git 专家级操作 | 复杂提交、变基策略、分支管理、版本控制 |

### 2.2 Oh-My-OpenCode 扩展技能

Oh-My-OpenCode(OMO) 作为 OpenCode 增强版，提供更多专业内置技能：

|----------|----------|----------|

| **`docker-optimize`** | Docker 镜像优化、多阶段构建建议 | Docker CLI、trivy |
| **`api-docs`** | API 文档生成、接口规范验证 | Swagger/OpenAPI 工具 |

## 三、自定义 Skill 规范

### 3.1 加载规则与存放目录

#### 3.1.1 三级加载优先级（从上到下，高优覆盖低优）

1. **项目级（最高优先级，推荐项目定制技能）**
   ```
   项目根目录/.opencode/skills/技能文件夹/SKILL.md
   ```

2. **全局用户级（个人通用技能，所有项目生效）**
   ```
   # Windows
   C:\Users\用户名\.config\opencode\skills\
   
   # Mac/Linux
   ~/.config/opencode/skills/
   ```

3. **自定义扩展路径**
   在 `opencode.json` 配置 `skills.paths` 追加自定义目录。

#### 3.1.2 强制目录规范

1. 每一个技能 = **独立文件夹**
2. 文件夹名 = 技能唯一 ID（命名规则：小写 + 短横线，如 `python-lint`、`git-flow`）
3. 文件夹内必须存在：**大写命名 SKILL.md**（硬性要求，小写无法识别）
4. 禁止同名技能：高优先级目录技能会直接覆盖低优先级同名技能

### 3.2 标准文件结构

完整文件分为两部分：
1. **YAML Frontmatter 头部元数据**（必填，控制技能基础信息）
2. **Markdown 指令正文**（行为约束、使用场景、执行流程、示例）

#### 3.2.1 完整元数据字段示例

```yaml
---
# 【必填】技能唯一标识，与文件夹名保持一致
name: python-code-standard
# 【必填】简短描述，用于 AI 匹配、技能列表展示
description: 统一 Python 代码编写规范、注释要求、格式检查流程
# 【可选】分类，用于技能管理、权限分组
category: develop
# 【可选】版本号
version: 1.0.0
# 【可选】开源协议
license: MIT
# 【可选】标签，提升语义匹配命中率
tags: ["python", "lint", "代码规范", "格式化"]
# 【可选】作用域：global/project
scope: project
---
```

#### 3.2.2 正文通用固定模块（官方推荐，AI 识别最优）

```markdown
## 能力说明
当前技能的核心作用、解决的问题、约束范围。

## 适用场景
- 场景 1：新建/修改 Python 代码
- 场景 2：代码重构、优化
- 场景 3：代码自查、规范整改

## 执行规则（核心约束）
1. 强制编码规范
2. 必须添加文件级/函数级注释
3. 禁止冗余代码、魔法数值
4. 输出代码必须符合 PEP8

## 操作步骤
1. 分析当前代码结构
2. 检查格式、命名、注释
3. 给出整改建议 + 修正后代码
4. 附加优化说明

## 示例参考
提供命令/代码/输出示例，降低 AI 理解成本
```

### 3.3 命名与语法规范

#### 3.3.1 文件夹/技能名
- 只能：小写字母、数字、短横线 `-`
- 禁止：大写、下划线、空格、中文、特殊符号
- 示例：`vue-component-gen`、`docker-build-opt`

#### 3.3.2 文件格式
- 技能主文件：必须 `SKILL.md` 大写
- 编码统一：UTF-8
- YAML 头部必须用 `---` 包裹，语法严格合规（禁止缩进错误）

#### 3.3.3 正文编写原则
- 指令化、短句化、条款式，少长段落
- 明确「强制要求」「禁止行为」「必须步骤」
- 增加关键词，提升自动触发命中率

## 四、YAML 头信息详解

### 4.1 标准格式

#### 4.1.1 最简标准格式（仅必填字段，推荐新手）

```markdown
---
name: 技能文件夹名
description: 技能功能描述（1-1024 字符）
---
```

#### 4.1.2 完整标准格式（所有官方支持字段）

```markdown
---
# 必填字段
name: skill-folder-name
description: 详细描述技能功能、作用、约束

# 可选字段（官方推荐）
category: 分类名
version: "1.0.0"
license: MIT
tags: ["标签 1", "标签 2", "标签 3"]
scope: project
author: 你的名字
---
```

### 4.2 字段全解

| 字段名 | 必填 | 格式要求 | 作用说明 |
|--------|------|----------|----------|
| `name` | ✅ 是 | 小写字母 + 数字 + 短横线，与文件夹同名 | 技能唯一 ID，手动调用/自动加载的标识 |
| `description` | ✅ 是 | 1~1024 字符，清晰简洁 | AI 语义匹配、自动触发的核心依据 |
| `category` | ❌ 否 | 英文/中文，如 `develop`/`git` | 技能分类，方便管理 |
| `version` | ❌ 否 | 字符串，如 `"1.0.0"` | 技能版本号 |
| `license` | ❌ 否 | 如 `MIT`/`Apache-2.0` | 开源协议 |
| `tags` | ❌ 否 | 字符串数组 `["标签 1","标签 2"]` | 关键词，大幅提升自动触发概率 |
| `scope` | ❌ 否 | `project`/`global` | 作用域：项目内生效/全局生效 |
| `author` | ❌ 否 | 文本 | 作者信息 |

### 4.3 语法铁律

1. **包裹符**
   头信息必须用 `---` 单独一行**开头**和**结尾**，前后不能有任何字符。

2. **键值对格式**
   `key: value`，**冒号后面必须加一个空格**，否则语法报错。

3. **缩进规则**
   只能用 **空格缩进**（2 个空格），**绝对不能用 Tab**。

4. **数组格式**
   `tags` 是数组，必须用 `[]` 包裹，字符串用双引号，逗号分隔。

5. **命名规则**
   `name` 必须和**技能文件夹名完全一致**（小写 + 短横线，无大写/空格/中文）。

6. **字符串格式**
   普通描述无需加引号，包含特殊符号（如 `:`、`-`）时用双引号包裹。

### 4.4 常见错误

1. ❌ 冒号后没加空格
   ```yaml
   name:my-skill  # 错误
   name: my-skill # 正确
   ```

2. ❌ 用 Tab 缩进
   YAML 只识别空格，Tab 会直接解析失败。

3. ❌ `name` 和文件夹名不一致
   文件夹：`go-spec`，YAML：`name: go-code-spec` → 不生效。

4. ❌ 包裹符 `---` 格式错误
   开头/结尾必须**单独一行**，不能和文字同行。

5. ❌ `tags` 格式错误
   ```yaml
   tags: golang, 规范 # 错误
   tags: ["golang", "规范"] # 正确
   ```

## 五、Skill 设计模式

### 5.1 五大核心模式

| 模式 | 解决的问题 | 核心技巧 |
|------|------------|----------|
| **01 知识注入型 (Tool Wrapper)** | AI 通用知识不够精准 | 将专业知识打包成 `references/` 下的文件（如 `conventions.md`、`best-practices.md`），仅在处理特定技术时按需加载，避免污染上下文 |
| **02 模板生成型 (Generator)** | 输出格式不一致、结构飘忽 | 在 `assets/` 中放置输出模板（骨架），在 `references/` 中定义风格指南，让 AI 按模板填内容而非自由发挥 |
| **03 审查打分型 (Reviewer)** | 需要按固定标准检查评估 | 将检查清单独立为 `review-checklist.md`（按严重级别组织），`SKILL.md` 只定义审查流程（加载→应用→报告），换标准只需换清单 |
| **04 反转采访型 (Inversion)** | AI 在信息不足时急于动手，导致偏差大 | 翻转交互方向：AI 先作为采访者，分阶段追问（问题是什么→约束与边界→信息齐了再生成），变量不齐不往下写 |
| **05 流水线型 (Pipeline)** | 任务复杂、步骤多，AI 容易跳步或遗漏 | 将任务拆分为严格阶段（如 Parse→Generate→Assemble→Verify），每个阶段设检查点（Gate），上一步未通过验证不允许进入下一步 |

> **组合原则**：五种模式可组合使用，先确定主模式，再嵌入其他模式。

### 5.2 模式选择决策树

1. 这个 Skill 是否产生输出？
   - 是 → 是否基于模板输出？
     - 是 → Generator
     - 否 → Tool Wrapper
   - 否 → 是否评估已有输入？
     - 是 → Reviewer
     - 否 → 是否需要先向用户收集信息？
       - 是 → Inversion
       - 否 → 是否有严格的顺序步骤？
         - 是 → Pipeline
         - 否 → Tool Wrapper

### 5.3 九大最佳实践

#### 内容层：写什么

1. **Description 是给模型看的** —— 写触发时机，而不是写摘要
2. **不陈述显而易见的事** —— 补充上下文和"坑"，不重复教材式内容
3. **构建 Gotchas 部分** —— 信号最强的是你踩过的坑，把实战经验写进去

#### 结构层：怎么组织

4. **文件系统 + 渐进式披露** —— Skill 是一个文件夹，按需加载，不要一次性全塞进上下文
5. **仔细考虑设置流程** —— 用 `config.json` 存储配置，与逻辑分离
6. **避免过度约束 AI** —— 约束目标（What），不约束路径（How）

#### 高级技术层：运行时优化

7. **内存与数据存储** —— 写日志，让 Skill 有记忆能力
8. **存储脚本并生成代码** —— 给 AI 可执行的代码/脚本，而不是只给文档说明
9. **按需 Hooks** —— 设置会话级护栏（Guardrails），用完即失效，避免长期污染

## 六、Skill 调用 Agent

### 6.1 内置 Agent 清单

OpenCode 固定可调用 Agent 名称（直接用这些名字调用，不能自定义）：

| Agent 名称 | 核心能力 |
|------------|----------|
| `code-agent` | 代码编写、重构、生成、修改（最常用） |
| `review-agent` | 代码审查、质量检测、漏洞扫描 |
| `git-agent` | Git 提交、分支、变基、版本管理 |
| `web-agent` | 网络搜索、API 文档查询 |
| `explore-agent` | 项目结构探索、依赖分析 |
| `docker-agent` | 容器构建、镜像优化 |
| `test-agent` | 单元测试、用例编写 |

### 6.2 三种调用方式

#### 6.2.1 方式 1：正文语义调用（⭐ 最常用、零配置）

直接在 `SKILL.md` 正文里，用**固定指令句式**指派 Agent，AI 会自动识别并加载对应 Agent 执行任务。

**万能调用句式**：
```markdown
## 执行指派（核心：调用 Agent）
1. 将当前任务 **委托给 code-agent 执行**
2. 代码审查环节 **自动加载 review-agent 完成校验**
3. 最终提交 **由 git-agent 负责标准化操作**
```

**极简示例**：
```markdown
---
name: code-publish-skill
description: 代码发布规范，调用审查+Git Agent 自动执行
---

## 强制规则
1. 所有代码必须通过 review-agent 审查
2. 提交操作必须由 git-agent 完成

## 执行步骤
1. 加载 code-agent 编写/修改代码
2. 委托 review-agent 做代码质量检查
3. 指派 git-agent 生成规范提交并推送
```

#### 6.2.2 方式 2：配置文件永久绑定

在项目 `opencode.json` 中，将**自定义 Skill 与 Agent 强绑定**，只要加载该 Skill，系统自动启动对应 Agent。

**配置模板**：
```json
{
  "skills": {
    "autoLoad": ["code-publish-skill"]
  },
  "agents": {
    // 技能绑定 Agent：skill 名称 → 对应执行 Agent
    "bindings": {
      "code-publish-skill": ["review-agent", "git-agent"]
    }
  }
}
```

#### 6.2.3 方式 3：对话内命令联动（调试专用）

调试时用命令，**手动加载 Skill + 切换 Agent**，快速验证调用效果：
```bash
# 1. 加载自定义 Skill
/use code-publish-skill

# 2. 切换到指定 Agent 执行
/agent use review-agent
```

### 6.3 多 Agent 流水线编排

复杂任务可以在 Skill 中设计**Agent 执行流水线**：

```markdown
## 全流程 Agent 编排
1. 启动 web-agent 搜索最新技术方案
2. 交付 code-agent 生成代码
3. 转交 test-agent 编写单元测试
4. 提交 review-agent 质量审查
5. 最后由 git-agent 完成发布
```

## 七、Skill 调用其他 Skill

### 7.1 三种标准调用方式

#### 7.1.1 方式 1：正文语义调用（⭐ 首选，零配置，最稳定）

直接在**主技能的 SKILL.md 正文**中，用**自然语言指令**声明「加载/启用/组合 XX 技能」，AI 会自动识别并加载依赖技能。

**万能调用句式**：
```markdown
1. 自动加载【skill-name】技能并执行
2. 配合【skill-name2】技能完成校验
3. 委托【skill-name3】技能处理后续流程
```

**完整示例**：
```markdown
---
name: project-full-publish
description: 项目全流程发布，自动调用代码审查 + 提交 + 构建技能
tags: ["发布", "自动化"]
---

## 能力说明
本技能为**编排主技能**，自动调用以下子技能完成全流程：
1. /review 代码审查技能
2. /commit 规范提交技能
3. docker-optimize 镜像优化技能

## 执行流程（固定顺序）
1. 调用 **review** 技能完成代码质量审查
2. 审查通过后，调用 **commit** 技能生成标准化提交
3. 最后调用 **docker-optimize** 技能优化容器镜像
4. 输出完整发布报告
```

#### 7.1.2 方式 2：配置文件自动绑定

在项目 `opencode.json` 中，将**主技能与依赖技能强绑定**，只要加载主技能，系统**自动加载所有子技能**。

**配置模板**：
```json
{
  "skills": {
    // 自动加载：主技能 + 所有需要调用的子技能
    "autoLoad": [
      "project-full-publish",  // 主技能
      "review",                // 子技能 1
      "commit",                // 子技能 2
      "docker-optimize"         // 子技能 3
    ]
  }
}
```

#### 7.1.3 方式 3：对话命令手动调用（调试专用）

在 OpenCode 对话中，直接用命令**同时加载多个技能**，快速验证调用效果：
```bash
# 加载主技能
/use project-full-publish

# 加载子技能（可叠加多个）
/use review
/use commit
/use docker-optimize

# 查看所有已加载技能（验证是否调用成功）
/skill list
```

### 7.2 技能编排器实战

**目录结构**：
```
.opencode/skills/
├── project-full-publish/   # 主技能（编排器）
│   └── SKILL.md
├── code-check-skill/       # 子技能 1
├── git-submit-skill/       # 子技能 2
└── build-deploy-skill/     # 子技能 3
```

**主技能 SKILL.md**：
```markdown
---
name: project-full-publish
description: 全流程自动化编排技能，调用代码检查+Git 提交 + 构建部署
category: automate
version: 1.0.0
---

## 调用的子技能清单
1. code-check-skill：代码规范检查
2. git-submit-skill：Git 标准化提交
3. build-deploy-skill：项目构建与部署

## 执行流水线
1. 先启用 **code-check-skill** 校验代码
2. 校验通过后，启用 **git-submit-skill** 提交代码
3. 最后启用 **build-deploy-skill** 完成构建部署
4. 所有技能必须按顺序执行，不可跳过
```

### 7.3 必守规则

1. **必须使用技能的 `name` 调用**
   不能用文件夹名/描述，必须用 YAML 头里的 `name` 字段。

2. **指令必须直白**
   用：`调用 `、` 加载 `、` 启用`、` 配合`、` 委托`，AI 识别率 100%。

3. **必须写明执行顺序**
   多技能调用一定要分 1、2、3 步骤，AI 会按顺序执行。

4. **技能名称全小写 + 短横线**
   禁止大写、中文、空格。

5. **无代码依赖**
   不需要导入、不需要路径，系统自动扫描技能。

## 八、实战案例

### 8.1 全自动多语言编码规范 Skill

**核心特性**：仅 1 个 Skill 文件、自动判断用户编辑的目标文件/语言、**按需加载对应规范**、无需手动选择、无需拆分多个子 Skill。

#### 8.1.1 核心设计思路

1. **单个 Skill 承载所有规范**：`auto-coding-spec`（无需多文件，极简维护）
2. **自动识别编辑目标**：通过**文件后缀、代码语言、编辑上下文**判断是 Python/Go/Node.js/C++
3. **精准按需加载**：识别后 **仅启用对应语言的规范**，屏蔽其他语言规则
4. **零手动操作**：AI 自动完成「识别→匹配→加载规范」全流程

#### 8.1.2 目录结构

```
你的项目根目录/
└── .opencode/
    └── skills/
        └── auto-coding-spec/  # 仅 1 个技能文件夹
            └── SKILL.md       # 仅 1 个技能文件（所有规范都在这里）
```

#### 8.1.3 完整 Skill 代码

```markdown
---
name: auto-coding-spec
description: 全自动多语言编码规范，自动识别编辑目标文件，按需加载 Python/Go/Node.js/C++官方编码规范，仅启用匹配语言的规则
category: develop
version: 1.0.0
tags: ["自动规范", "python", "go", "nodejs", "cpp", "代码检查", "按需加载"]
scope: project
---

# 🔍 核心能力（全自动执行）
1. 自动识别用户当前**编辑的目标文件/代码语言**
2. 仅加载**匹配语言**的编码规范，屏蔽其他无关规则
3. 严格按照对应语言的官方标准约束代码编写、审查、重构
4. 无识别目标时，不加载任何语言规范

# 📌 自动语言识别规则（AI 严格执行）
根据编辑目标的**文件后缀**判断语言：
- `.py` → Python 规范
- `.go` → Go 规范
- `.js` / `.ts` → Node.js/JavaScript 规范
- `.cpp` / `.h` / `.hpp` → C++ 规范

# 🚀 执行流程（固定不可修改）
1. 读取用户当前编辑的文件/代码上下文
2. 匹配文件后缀，确定编程语言
3. **仅启用对应语言的规范模块**
4. 按规范生成/检查/优化代码
5. 禁止启用未匹配语言的规范

# ======================
# 【按需加载：语言规范模块】
# 仅匹配的语言会被 AI 启用，其他模块自动忽略
# ======================

## 1. Python 规范（仅 .py 文件启用）
### 强制规则
- 缩进：4 个空格，禁止 Tab
- 行宽：88 字符（PEP8 标准）
- 命名：变量/函数小写 + 下划线，类大驼峰，常量全大写
- 必须写函数/类 docstring 注释
- 导入分组：标准库 → 第三方库 → 本地库
- 禁止裸 except，必须显式处理异常

## 2. Go 规范（仅 .go 文件启用）
### 强制规则
- 包名：全小写，无驼峰/下划线
- 缩进：Tab（Go 官方标准）
- 命名：导出标识符大写开头，驼峰命名
- 错误必须显式返回，禁止忽略 error
- 标准目录：internal / pkg / cmd
- 禁止业务代码使用 panic

## 3. Node.js/JavaScript 规范（仅 .js/.ts 文件启用）
### 强制规则
- 缩进：2 个空格
- 命名：变量/函数小驼峰，类大驼峰，常量全大写
- 使用 ES6+：const/let 替代 var，优先 async/await
- 统一模块化规范，禁止混用 CommonJS/ESM
- 禁止使用 var、eval、with 等危险语法

## 4. C++ 规范（仅 .cpp/.h/.hpp 文件启用）
### 强制规则
- 缩进：4 个空格
- 命名：类/结构体大驼峰，函数小驼峰，常量全大写
- 优先 C++11+ 标准，使用智能指针
- 禁止裸指针、内存泄漏
- 头文件使用 #pragma once 保护
- 函数输入参数用 const 引用

# ======================
# 【禁用规则】
# ======================
1. 未匹配的语言规范 **绝对禁止启用**
2. 不允许混合使用多种语言的规范
3. 必须严格遵循识别后的单一语言规范
```

#### 8.1.4 核心工作原理

1. **你编辑文件时**：AI 读取文件后缀（如 `main.py`）
2. **Skill 自动判断**：匹配 Python 规范
3. **按需加载**：**只启用 Python 规则**，忽略 Go/Node.js/C++ 所有内容
4. **代码约束**：生成的代码 100% 符合 Python 官方规范

#### 8.1.5 使用方式

**全自动使用（推荐）**：
```bash
# 1. 加载技能
/use auto-coding-spec

# 2. 直接编辑代码（如写 .go 文件）
# 3. AI 自动识别 → 自动加载 Go 规范
```

**永久自动加载（项目默认启用）**：
```json
{
  "skills": {
    "autoLoad": ["auto-coding-spec"]
  }
}
```

#### 8.1.6 扩展能力

直接在 Skill 中**新增模块**即可，无需修改结构：
```markdown
## 5. Java 规范（仅 .java 文件启用）
### 强制规则
- 缩进：4 个空格
- 命名：大驼峰类名，小驼峰方法名
...
```

## 附录

### A. 最佳实践

#### A.1 技能组织策略

- **按领域分类**：`git-*`、`docker-*`、`security-*` 等
- **按粒度拆分**：一个技能只解决一个核心问题，复杂流程拆分为多个子技能
- **项目 vs 全局**：
  - 项目级：存储项目特有流程与规范
  - 全局级：存储个人通用工作流与最佳实践

#### A.2 编写高质量 Skill 的 6 步打磨法

1. **明确目标**：确定技能解决的具体问题与使用场景
2. **收集知识**：整理领域最佳实践、步骤、示例
3. **结构化内容**：遵循"What I do→When to use→Steps→Examples"结构
4. **精简语言**：使用 AI 易理解的明确指令，避免模糊表述
5. **测试优化**：在实际任务中测试，根据反馈调整
6. **版本迭代**：持续优化技能内容，添加新场景与示例

### B. 常见问题排查

| 问题 | 排查步骤 |
|------|----------|
| 技能不显示 | 检查文件名是否为 `SKILL.md`(大写)、目录名与技能名一致、元数据完整 |
| 权限被拒绝 | 检查 `opencode.json` 中的权限配置、代理是否禁用了 skill 工具 |
| 重复技能 | 搜索所有技能目录，删除重复名称技能 (按发现顺序加载第一个) |
| 解析错误 | 验证 YAML 元数据语法，确保格式正确 |
| 技能调用失败 | 确认使用技能的 `name` 字段调用、指令直白、写明执行顺序 |

### C. Skill 使用方法总结

1. **自动调用**：代理根据任务描述自动匹配并加载合适技能
2. **手动调用**：使用`/skill use <skill-name>` 命令强制使用特定技能
3. **委托任务**(OMO)：
   ```
   delegate_task(
     "提交当前更改，遵循提交规范",
     load_skills=["git-master", "commit"]
   )
   ```

### D. 权限配置示例

```json
{
  "permission": {
    "skill": {
      "git-*": "allow",  // 允许所有 git 相关技能
      "internal-*": "deny",  // 拒绝内部技能
      "*": "ask"  // 其他技能询问用户
    }
  }
}
```
