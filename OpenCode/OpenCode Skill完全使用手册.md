## 一、Skill 核心概念

### 1.1 定义与本质

Skill 是**带 YAML 元数据的 Markdown 文件**，存储在特定目录中，包含：
- **元数据 (Frontmatter)**：技能名称、描述、许可证等
- **内容主体**：功能说明、使用场景、步骤、示例等

**核心价值**：将专业知识与工作流程"封装"，实现**一次编写、多次复用**，解决 AI 输出不一致、流程不规范等问题。

### 1.2 关键特性

| 特性 | 说明 |
|------|------|
| **按需加载** | 代理根据任务自动选择或用户手动调用 |
| **权限控制** | 支持全局/代理级权限，可`allow`/`deny`/`ask` |
| **多级发现** | 项目级→全局级→自定义路径，优先级递减 |
| **无参数设计** | 当前版本技能无参数，提供静态指令 (未来将支持) |
| **跨平台兼容** | 支持 Claude Desktop 技能格式 |

## 三、自定义 Skill 规范

### 3.1 加载规则与存放目录

#### 3.1.1 三级加载优先级（从上到下，高优覆盖低优）

1. **项目级（最高优先级，推荐项目定制技能）**
   ```
   项目根目录/.opencode/skills/技能文件夹/SKILL.md
   ```

2. **全局用户级（个人通用技能，所有项目生效）**
   ```
   # Windows
   C:\Users\用户名\.config\opencode\skills\
   
   # Mac/Linux
   ~/.config/opencode/skills/
   ```

3. **自定义扩展路径**
   在 `opencode.json` 配置 `skills.paths` 追加自定义目录。

#### 3.1.2 强制目录规范

1. 每一个技能 = **独立文件夹**
2. 文件夹名 = 技能唯一 ID（命名规则：小写 + 短横线，如 `python-lint`、`git-flow`）
3. 文件夹内必须存在：**大写命名 SKILL.md**（硬性要求，小写无法识别）
4. 禁止同名技能：高优先级目录技能会直接覆盖低优先级同名技能

### 3.2 标准文件结构

完整文件分为两部分：
1. **YAML Frontmatter 头部元数据**（必填，控制技能基础信息）
2. **Markdown 指令正文**（行为约束、使用场景、执行流程、示例）

#### 3.2.1 完整元数据字段示例

```yaml
---
# 【必填】技能唯一标识，与文件夹名保持一致
name: python-code-standard
# 【必填】简短描述，用于 AI 匹配、技能列表展示
description: 统一 Python 代码编写规范、注释要求、格式检查流程
# 【可选】分类，用于技能管理、权限分组
category: develop
# 【可选】版本号
version: 1.0.0
# 【可选】开源协议
license: MIT
# 【可选】标签，提升语义匹配命中率
tags: ["python", "lint", "代码规范", "格式化"]
# 【可选】作用域：global/project
scope: project
---
```

**字段全解**

| 字段名 | 必填 | 格式要求 | 作用说明 |
|--------|------|----------|----------|
| `name` | ✅ 是 | 小写字母 + 数字 + 短横线，与文件夹同名 | 技能唯一 ID，手动调用/自动加载的标识 |
| `description` | ✅ 是 | 1~1024 字符，清晰简洁 | AI 语义匹配、自动触发的核心依据 |
| `category` | ❌ 否 | 英文/中文，如 `develop`/`git` | 技能分类，方便管理 |
| `version` | ❌ 否 | 字符串，如 `"1.0.0"` | 技能版本号 |
| `license` | ❌ 否 | 如 `MIT`/`Apache-2.0` | 开源协议 |
| `tags` | ❌ 否 | 字符串数组 `["标签 1","标签 2"]` | 关键词，大幅提升自动触发概率 |
| `scope` | ❌ 否 | `project`/`global` | 作用域：项目内生效/全局生效 |
| `author` | ❌ 否 | 文本 | 作者信息 |

**语法铁律**

1. **包裹符**
   头信息必须用 `---` 单独一行**开头**和**结尾**，前后不能有任何字符。

2. **键值对格式**
   `key: value`，**冒号后面必须加一个空格**，否则语法报错。

3. **缩进规则**
   只能用 **空格缩进**（2 个空格），**绝对不能用 Tab**。

4. **数组格式**
   `tags` 是数组，必须用 `[]` 包裹，字符串用双引号，逗号分隔。

5. **命名规则**
   `name` 必须和**技能文件夹名完全一致**（小写 + 短横线，无大写/空格/中文）。

6. **字符串格式**
   普通描述无需加引号，包含特殊符号（如 `:`、`-`）时用双引号包裹。


#### 3.2.2 正文通用固定模块（官方推荐，AI 识别最优）

```markdown
## 能力说明
当前技能的核心作用、解决的问题、约束范围。

## 适用场景
- 场景 1：新建/修改 Python 代码
- 场景 2：代码重构、优化
- 场景 3：代码自查、规范整改

## 执行规则（核心约束）
1. 强制编码规范
2. 必须添加文件级/函数级注释
3. 禁止冗余代码、魔法数值
4. 输出代码必须符合 PEP8

## 操作步骤
1. 分析当前代码结构
2. 检查格式、命名、注释
3. 给出整改建议 + 修正后代码
4. 附加优化说明

## 示例参考
提供命令/代码/输出示例，降低 AI 理解成本
```

### 3.3 命名与语法规范

#### 3.3.1 文件夹/技能名
- 只能：小写字母、数字、短横线 `-`
- 禁止：大写、下划线、空格、中文、特殊符号
- 示例：`vue-component-gen`、`docker-build-opt`

#### 3.3.2 文件格式
- 技能主文件：必须 `SKILL.md` 大写
- 编码统一：UTF-8
- YAML 头部必须用 `---` 包裹，语法严格合规（禁止缩进错误）

#### 3.3.3 正文编写原则
- 指令化、短句化、条款式，少长段落
- 明确「强制要求」「禁止行为」「必须步骤」
- 增加关键词，提升自动触发命中率

## 五、Skill 设计模式

### 一、五种核心 Skill 设计模式

| 模式名称 | 核心解决问题 | 核心思路与实现 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **知识注入型 (Tool Wrapper)** | AI 通用知识不精准，需要注入领域/私有知识 | 1. 在 `SKILL.md` 中定义触发条件<br>2. 知识文件（如 `conventions.md`）存放在 `references/` 目录<br>3. 触发时按需加载，不污染无关上下文 | 前端 React 规范、数据库 SQL 规范、企业内部开发标准等 |
| **模板生成型 (Generator)** | AI 输出格式不统一、结构飘忽不定 | 1. `assets/` 目录存放输出模板（如 `report-template.md`）<br>2. `references/` 存放风格指南（如语气、命名规则）<br>3. 强制 AI 按模板结构填充内容，保证一致性 | API 文档生成、测试报告、周报/日报、技术方案模板等 |
| **审查打分型 (Reviewer)** | 需要 AI 按固定标准检查/评估内容 | 1. `SKILL.md` 定义审查流程：加载清单→逐项检查→按严重程度分类→生成报告<br>2. `references/` 存放检查清单（如 `review-checklist.md`）<br>3. 流程与规则解耦，换清单即可切换审查标准 | 代码规范审查、安全漏洞检查、文档合规性校验、设计评审等 |
| **反转采访型 (Inversion)** | AI 信息不足时盲目执行，结果偏差大 | 1. 流程反转：先提问，再执行<br>2. 分三阶段：<br>① 发现阶段：问清问题背景、用户、规模<br>② 约束阶段：确认平台、技术栈、需求边界<br>③ 合成阶段：信息齐全后再生成结果 | 客服机器人方案设计、产品需求拆解、系统架构规划等 |
| **流水线型 (Pipeline)** | 复杂任务 AI 易跳步/遗漏关键环节 | 1. 任务拆分为严格的阶段，每个阶段定义明确的输入、输出和检查点<br>2. 阶段间设置"关卡（Gate）"，上一步未通过验证无法进入下一步 | API 文档自动生成、代码重构、数据处理流程、多步骤自动化测试等 |

> 💡 关键提示：五种模式可**组合使用**，建议先确定主模式，再嵌入其他模式（比如流水线的某个步骤，可嵌入模板生成型输出）。

### 二、Skill 模式选型决策指南

用这个流程图快速判断该用哪种模式：

1. **先判断：Skill 是否要生成输出？**
   - ✅ 是 → 再判断：是否需要固定模板？
     - 是 → 用 **模板生成型（Generator）**
     - 否 → 用 **知识注入型（Tool Wrapper）**
   - ❌ 否 → 再判断：是否要评估已有输入？
     - 是 → 用 **审查打分型（Reviewer）**
     - 否 → 再判断：是否需要用户先输入信息？
       - 是 → 用 **反转采访型（Inversion）**
       - 否 → 再判断：是否有固定顺序的步骤？
         - 是 → 用 **流水线型（Pipeline）**
         - 否 → 用 **知识注入型（Tool Wrapper）**

### 三、5 种 Skill 设计模板 + 案例

以下是可直接复制使用的 5 种 Skill 设计模板 + 案例对照，按"先套模板，再填案例"的方式整理：

#### 01 知识注入型 (Tool Wrapper)

##### 通用模板

```markdown
## [Skill 名称]

### 背景
你是 [领域] 的 [身份]，拥有 [X] 年经验。[一句话说明为什么通用 AI 做不好这件事]。

### 工作方式
1. 用户提供 [输入内容]
2. **自动加载** `references/[知识文件].md`
3. 严格按照 [知识文件] 中的 [checklist/标准/规则] 执行
4. 输出：[具体交付物]

### 输出格式
- [格式要求 1]
- [格式要求 2]
- 必须引用 [知识文件] 中的 [条目编号/规则名称]

### 禁止事项
- [禁止自由发挥的点 1]
- [禁止跳过/忽略的点 2]
```

##### 案例：跨境电商选品助手

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[领域]` | 跨境电商 |
| `[身份]` | 选品专家 |
| `[X] 年经验` | 8 年亚马逊运营 |
| `[输入内容]` | 一个产品 idea 或竞品链接 |
| `[知识文件]` | `sourcing-criteria.md`（含市场/利润/供应链/合规/运营 5 大维度 checklist） |
| `[交付物]` | 通过/不通过判定 + 综合评分 + 风险等级 + 改进建议 |
| `[禁止事项]` | 不允许跳过任何评估维度；必须引用 checklist 条目编号 |

#### 02 模板生成型 (Generator)

##### 通用模板

```markdown
## [Skill 名称]

### 背景
[使用场景说明，强调"格式一致性"的重要性]。

### 工作方式
1. 用户提供 [原始数据/内容]
2. **强制使用** `assets/[模板文件].[扩展名]` 的格式
3. 将用户提供的内容"填入"模板对应位置
4. **禁止**：调整模板结构、增减章节、改变标题层级、自由发挥叙述

### 数据映射规则
- 用户说 "[原始表述示例]" → 填入 `[模板字段名]`
- 用户提到 "[原始表述示例]" → 填入 `[模板字段名]`
- [无数据时的处理方式]

### 输出要求
- 直接输出填充后的完整模板
- 不要输出"以下是按照模板生成的..."等废话
- 模板中无数据的字段，保留字段名并标注"[占位提示]"
```

##### 案例：投资人周报生成器

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[使用场景]` | 创始人每周五向天使投资人发送经营数据周报，投资人时间宝贵，必须 30 秒内抓住重点 |
| `[模板文件]` | `weekly-report-template.md`（含核心指标表/关键进展/风险阻塞/下周动作/资金状况 5 大固定章节） |
| `[原始表述→字段映射]` | "这周新增用户大概 500 多" → `周新增用户` 字段；"服务器挂了 2 小时" → `技术事故` 子项 |
| `[无数据处理]` | 保留字段名并标注"暂无数据" |

#### 03 审查打分型 (Reviewer)

##### 通用模板

```markdown

### 背景
你是 [身份]，需要对 [审查对象] 进行 [审查目的]。

2. **逐项检查**：对照 checklist 逐条验证

3. **分级定责**：
   - **P0（阻塞）**：[定义] → [处理方式]
   - **P1（严重）**：[定义] → [处理方式]
   - **P2（一般）**：[定义] → [处理方式]
   - **P3（建议）**：[定义] → [处理方式]

4. **输出结构化报告**

### 输出格式
#### 审查摘要
- 对象：
- 时间：
- **综合评分**：_ / 100（[扣分规则]）
- **审查结论**：[结论选项]

#### 问题清单（按严重程度排序）
| 编号 | 严重程度 | 检查项来源 | 问题描述 | 位置 | 修复建议 |

#### 正向反馈（至少 [N] 条）
```

##### 案例：代码审查官

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[审查对象]` | 提交的代码 |
| `[分类条件]` | React/Vue 前端 → `frontend-checklist.md`；涉及 API/数据库 → 同时加载 `security-checklist.md` |
| `[P0 定义]` | 安全漏洞、数据丢失风险、核心功能崩溃 |
| `[P1 定义]` | 性能明显劣化、可维护性差、不符合架构规范 |
| `[扣分规则]` | P0 每条扣 30 分，P1 扣 15 分，P2 扣 5 分，P3 扣 1 分 |

### 6.2 三种调用方式

#### 6.2.1 方式 1：正文语义调用（⭐ 最常用、零配置）

直接在 `SKILL.md` 正文里，用**固定指令句式**指派 Agent，AI 会自动识别并加载对应 Agent 执行任务。

**万能调用句式**：
```markdown
## 执行指派（核心：调用 Agent）
1. 将当前任务 **委托给 code-agent 执行**
2. 代码审查环节 **自动加载 review-agent 完成校验**
3. 最终提交 **由 git-agent 负责标准化操作**
```

**极简示例**：
```markdown
---
name: code-publish-skill
description: 代码发布规范，调用审查+Git Agent 自动执行
---

## 强制规则
1. 所有代码必须通过 review-agent 审查
2. 提交操作必须由 git-agent 完成

## 执行步骤
1. 加载 code-agent 编写/修改代码
2. 委托 review-agent 做代码质量检查
3. 指派 git-agent 生成规范提交并推送
```

#### 6.2.2 方式 2：配置文件永久绑定

在项目 `opencode.json` 中，将**自定义 Skill 与 Agent 强绑定**，只要加载该 Skill，系统自动启动对应 Agent。

```json
    "autoLoad": ["code-publish-skill"]

  "agents": {

}
```

#### 6.2.3 方式 3：对话内命令联动（调试专用）

## 八、实战案例

### 8.1 全自动多语言编码规范 Skill

**核心特性**：仅 1 个 Skill 文件、自动判断用户编辑的目标文件/语言、**按需加载对应规范**、无需手动选择、无需拆分多个子 Skill。

#### 8.1.1 核心设计思路

1. **单个 Skill 承载所有规范**：`auto-coding-spec`（无需多文件，极简维护）
2. **自动识别编辑目标**：通过**文件后缀、代码语言、编辑上下文**判断是 Python/Go/Node.js/C++
3. **精准按需加载**：识别后 **仅启用对应语言的规范**，屏蔽其他语言规则
4. **零手动操作**：AI 自动完成「识别→匹配→加载规范」全流程

#### 8.1.2 目录结构

```
你的项目根目录/
└── .opencode/
    └── skills/
        └── auto-coding-spec/  # 仅 1 个技能文件夹
            └── SKILL.md       # 仅 1 个技能文件（所有规范都在这里）
```

#### 8.1.3 完整 Skill 代码

```markdown
---
name: auto-coding-spec
description: 全自动多语言编码规范，自动识别编辑目标文件，按需加载 Python/Go/Node.js/C++官方编码规范，仅启用匹配语言的规则
category: develop
version: 1.0.0
tags: ["自动规范", "python", "go", "nodejs", "cpp", "代码检查", "按需加载"]
scope: project
---

# 🔍 核心能力（全自动执行）
1. 自动识别用户当前**编辑的目标文件/代码语言**
2. 仅加载**匹配语言**的编码规范，屏蔽其他无关规则
3. 严格按照对应语言的官方标准约束代码编写、审查、重构
4. 无识别目标时，不加载任何语言规范

# 📌 自动语言识别规则（AI 严格执行）
根据编辑目标的**文件后缀**判断语言：
- `.py` → Python 规范
- `.go` → Go 规范
- `.js` / `.ts` → Node.js/JavaScript 规范
- `.cpp` / `.h` / `.hpp` → C++ 规范

# 🚀 执行流程（固定不可修改）
1. 读取用户当前编辑的文件/代码上下文
2. 匹配文件后缀，确定编程语言
3. **仅启用对应语言的规范模块**
4. 按规范生成/检查/优化代码
5. 禁止启用未匹配语言的规范

# ======================
# 【按需加载：语言规范模块】
# 仅匹配的语言会被 AI 启用，其他模块自动忽略
# ======================

## 1. Python 规范（仅 .py 文件启用）
### 强制规则
- 缩进：4 个空格，禁止 Tab
- 行宽：88 字符（PEP8 标准）
- 命名：变量/函数小写 + 下划线，类大驼峰，常量全大写
- 必须写函数/类 docstring 注释
- 导入分组：标准库 → 第三方库 → 本地库
- 禁止裸 except，必须显式处理异常

## 2. Go 规范（仅 .go 文件启用）
### 强制规则
- 包名：全小写，无驼峰/下划线
- 缩进：Tab（Go 官方标准）
- 命名：导出标识符大写开头，驼峰命名
- 错误必须显式返回，禁止忽略 error
- 标准目录：internal / pkg / cmd
- 禁止业务代码使用 panic

## 3. Node.js/JavaScript 规范（仅 .js/.ts 文件启用）
### 强制规则
- 缩进：2 个空格
- 命名：变量/函数小驼峰，类大驼峰，常量全大写
- 使用 ES6+：const/let 替代 var，优先 async/await
- 统一模块化规范，禁止混用 CommonJS/ESM
- 禁止使用 var、eval、with 等危险语法

## 4. C++ 规范（仅 .cpp/.h/.hpp 文件启用）
### 强制规则
- 缩进：4 个空格
- 命名：类/结构体大驼峰，函数小驼峰，常量全大写
- 优先 C++11+ 标准，使用智能指针
- 禁止裸指针、内存泄漏
- 头文件使用 #pragma once 保护
- 函数输入参数用 const 引用

# ======================
# 【禁用规则】
# ======================
1. 未匹配的语言规范 **绝对禁止启用**
2. 不允许混合使用多种语言的规范
3. 必须严格遵循识别后的单一语言规范
```

#### 8.1.4 核心工作原理

1. **你编辑文件时**：AI 读取文件后缀（如 `main.py`）
2. **Skill 自动判断**：匹配 Python 规范
3. **按需加载**：**只启用 Python 规则**，忽略 Go/Node.js/C++ 所有内容
4. **代码约束**：生成的代码 100% 符合 Python 官方规范

```json

    "autoLoad": ["auto-coding-spec"]
  }

```

### 四、使用技巧与最佳实践

#### 1. 技能管理命令
```bash
## 列出所有已加载技能
/skill list

## 搜索可用技能
skill_find query="git"

## 加载技能
/use 技能名称

## 卸载技能
/skill remove 技能名称
```

#### 2. 按需加载原则
- 开发Python项目：只加载`python-coding-spec`和`git-master`，避免冗余
- 写文档时：加载`docx-word`和`doc-coauthoring`，专注内容创作
- 多语言项目：使用之前设计的`auto-coding-spec`，自动识别语言

#### 3. 安全与性能建议
- 优先选择GitHub高星、社区活跃的技能，避免恶意代码
- 定期更新技能，保持功能最新
- 不常用的技能及时卸载，减少内存占用

开源OpenCode Skill资源丰富，从官方渠道、社区市场到GitHub都能找到高质量技能。工作中优先使用开发、办公类技能提升效率，生活中用学习、工具类技能改善体验。记住**按需加载**原则，让AI助手专注于当前任务，发挥最大价值。

## 附录

### A. 最佳实践

#### A.1 技能组织策略

- **按领域分类**：`git-*`、`docker-*`、`security-*` 等
- **按粒度拆分**：一个技能只解决一个核心问题，复杂流程拆分为多个子技能
- **项目 vs 全局**：
  - 项目级：存储项目特有流程与规范
  - 全局级：存储个人通用工作流与最佳实践

#### A.2 编写高质量 Skill 的 6 步打磨法

1. **明确目标**：确定技能解决的具体问题与使用场景
2. **收集知识**：整理领域最佳实践、步骤、示例
3. **结构化内容**：遵循"What I do→When to use→Steps→Examples"结构
4. **精简语言**：使用 AI 易理解的明确指令，避免模糊表述
5. **测试优化**：在实际任务中测试，根据反馈调整
6. **版本迭代**：持续优化技能内容，添加新场景与示例

### B. 常见问题排查

| 问题 | 排查步骤 |
|------|----------|
| 技能不显示 | 检查文件名是否为 `SKILL.md`(大写)、目录名与技能名一致、元数据完整 |
| 权限被拒绝 | 检查 `opencode.json` 中的权限配置、代理是否禁用了 skill 工具 |
| 重复技能 | 搜索所有技能目录，删除重复名称技能 (按发现顺序加载第一个) |
| 解析错误 | 验证 YAML 元数据语法，确保格式正确 |
| 技能调用失败 | 确认使用技能的 `name` 字段调用、指令直白、写明执行顺序 |

### C. Skill 使用方法总结

1. **自动调用**：代理根据任务描述自动匹配并加载合适技能
2. **手动调用**：使用`/skill use <skill-name>` 命令强制使用特定技能
3. **委托任务**(OMO)：
   ```
   delegate_task(
     "提交当前更改，遵循提交规范",
     load_skills=["git-master", "commit"]
   )
   ```

### D. 权限配置示例

```json
{
  "permission": {
    "skill": {
      "git-*": "allow",  // 允许所有 git 相关技能
      "internal-*": "deny",  // 拒绝内部技能
      "*": "ask"  // 其他技能询问用户
    }
  }
}
```
