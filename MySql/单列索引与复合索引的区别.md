# 单列索引与复合索引的区别

## 1. 基本概念

| 索引类型 | 定义 | 示例 |
|---|---|---|
| **单列索引** | 只包含一个字段的索引 | `INDEX idx_TradeDate(TradeDate)` |
| **复合索引** | 包含多个字段的索引 | `INDEX idx_TradeDate_SendState(TradeDate, SendState)` |

---

## 2. 为什么复合索引能优化组合查询？

### 场景：查询 `TradeDate = 20240101 AND SendState = 2`

```sql
SELECT * FROM sendorderlist 
WHERE TradeDate = 20240101 AND SendState = 2;
```

### 使用单列索引的情况

```sql
-- 只有两个单列索引
INDEX idx_TradeDate(TradeDate)
INDEX idx_SendState(SendState)
```

**执行过程：**
1. MySQL 只能选择**其中一个索引**（通常选选择性更高的）
2. 假设选择 `idx_TradeDate`，先找出所有 `TradeDate = 20240101` 的记录
3. 然后**逐行过滤** `SendState = 2` 的条件

```
┌─────────────────────────────────────────────────────────────┐
│  全表数据  →  idx_TradeDate 查找  →  1000 行 TradeDate 匹配   │
│                                                         ↓    │
│  逐行检查 SendState → 最终得到 50 行结果                      │
└─────────────────────────────────────────────────────────────┘
```

### 使用复合索引的情况

```sql
-- 复合索引
INDEX idx_TradeDate_SendState(TradeDate, SendState)
```

**执行过程：**
1. 直接在复合索引中定位 `(TradeDate=20240101, SendState=2)` 的组合
2. **一步到位**获取精确结果

```
┌─────────────────────────────────────────────────────────────┐
│  idx_TradeDate_SendState 查找  →  直接得到 50 行最终结果       │
└─────────────────────────────────────────────────────────────┘
```

---

## 3. 复合索引的"最左前缀原则"

复合索引 `(A, B, C)` 可以用于以下查询：

| 查询条件 | 能否使用索引 | 说明 |
|---|---|---|
| `WHERE A = ?` | ✅ 可以 | 使用第 1 列 |
| `WHERE A = ? AND B = ?` | ✅ 可以 | 使用前两列 |
| `WHERE A = ? AND B = ? AND C = ?` | ✅ 可以 | 使用全部三列 |
| `WHERE B = ?` | ❌ 不可以 | 跳过第 1 列 |
| `WHERE B = ? AND C = ?` | ❌ 不可以 | 跳过第 1 列 |
| `WHERE A = ? AND C = ?` | ⚠️ 部分 | 只用 A 列，C 列需行过滤 |

**示例：**
```sql
-- 索引：idx_A_B_C (A, B, C)

-- ✅ 可以使用索引
SELECT * FROM table WHERE A = 1;
SELECT * FROM table WHERE A = 1 AND B = 2;
SELECT * FROM table WHERE A = 1 AND B = 2 AND C = 3;

-- ❌ 无法使用索引（跳过第 1 列）
SELECT * FROM table WHERE B = 2;
SELECT * FROM table WHERE B = 2 AND C = 3;

-- ⚠️ 部分使用索引（C 列无法利用索引）
SELECT * FROM table WHERE A = 1 AND C = 3;
```

## 4. 实例分析

### 原查询（sp_monitor_realorders_status.sql）
```sql
WHERE l.TradeDate = @v_TodayStr      -- 第 1 列
  AND l.SendState = 2                -- 第 2 列
  AND r.RunStatus = 1                -- 关联表字段
  AND r.StartStatus = 1
```

### 索引选择对比

| 方案 | 索引 | 效率 |
|---|---|---|
| 单列索引 | `idx_TradeDate`, `idx_SendState` | 中等 - MySQL 只能选一个 |
| 复合索引 | `idx_TradeDate_SendState(TradeDate, SendState)` | 高 - 一步定位 |

---

## 5. 何时不需要复合索引？

1. **数据量小**：表只有几万行，单列索引已足够
2. **选择性低**：字段区分度不高（如 `SendState` 只有 1,2,3,4 几个值）
3. **写入频繁**：复合索引会增加 INSERT/UPDATE 的开销
4. **已有单列索引可用**：MySQL 可以进行 **索引合并（Index Merge）**

### 索引合并（Index Merge）

当存在多个单列索引时，MySQL 可以同时使用它们：

```sql
-- 查询
SELECT * FROM sendorderlist 
WHERE TradeDate = 20240101 AND SendState = 2;

-- 执行计划可能显示：
-- Using intersect(idx_TradeDate, idx_SendState)
```

但索引合并的效率通常不如复合索引。

## 6. 实际应用建议

### 当前存储过程的索引建议

```sql
-- ✅ 建议添加（数据量大时）
ALTER TABLE simulate_plan_order ADD INDEX idx_ExecDate_BSFlag (ExecDate, BSFlag3);
-- 原因：ExecDate 和 BSFlag3 经常一起出现在 WHERE 条件中

-- ⚠️ 可选添加（视 sendorderlist 数据量而定）
-- ALTER TABLE sendorderlist ADD INDEX idx_TradeDate_SendState (TradeDate, SendState);
-- 原因：已有单列索引，如果数据量不大则无需复合索引
```

### 完整索引建议

```sql
-- simulate_plan_order 表添加索引（2 字段）
ALTER TABLE simulate_plan_order ADD INDEX idx_ExecDate_BSFlag (ExecDate, BSFlag3);
ALTER TABLE simulate_plan_order ADD INDEX idx_PlanId_ExecDate (PlanId, ExecDate);

-- run_realtrade 表添加索引（2 字段）

-- simulate_finish_state 表添加索引（2 字段）
ALTER TABLE simulate_finish_state ADD INDEX idx_RiskDate_EndDate (RiskDate, EndDate2);
```

## 7. 总结对比

| 特性 | 单列索引 | 复合索引 |
|---|---|---|
| **字段数量** | 1 个 | 多个（通常 2-3 个） |
| **适用场景** | 单字段查询 | 多字段组合查询 |
| **查询效率** | 中等 | 高（对于组合条件） |
| **写入开销** | 低 | 中等（每个索引都需维护） |
| **存储空间** | 小 | 中等 |
| **灵活性** | 高 | 需遵循最左前缀原则 |
| **维护成本** | 低 | 中等 |

---

## 8. 最佳实践

1. **优先使用单列索引**：满足 80% 的查询场景
2. **针对性添加复合索引**：只针对频繁出现且性能瓶颈明显的组合查询
3. **遵循最左前缀原则**：设计复合索引时，将选择性高的字段放在前面
4. **避免过度索引**：每个索引都有维护成本，写入频繁的表尤需谨慎
5. **定期分析索引使用情况**：使用 `SHOW INDEX` 和 `EXPLAIN` 检查索引效果
