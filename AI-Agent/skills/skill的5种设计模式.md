# 五种常见 AI Skill 设计方向解读

## 五种 Skill 设计模式

| 编号 | 类型 | 解决的问题 | 核心思路 |
| ---- | ---- | ---------- | -------- |
| 01 | 知识注入型 (Tool Wrapper) | AI 通用知识不够精准，需要注入"只有你知道的"专业知识 | 把专业经验打包成 `references` 文件，AI 需要时自动加载，按你的标准来做事 |
| 02 | 模板生成型 (Generator) | AI 每次生成的输出格式不一致，结构飘忽不定 | 在 `assets/` 里放一个输出模板，让 AI 按模板结构往里填内容，而不是自由发挥 |
| 03 | 审查打分型 (Reviewer) | 需要 AI 按照一套固定标准去检查、评估某个东西 | 把检查清单放在 `references/` 里，`SKILL.md` 定义审查流程：加载清单 → 逐项检查 → 按严重程度分类 → 输出结构化报告 |
| 04 | 反转采访型 (Inversion) | AI 总是在信息不足的情况下急着动手，结果偏差很大 | 翻转交互方向——不是用户提需求 AI 马上执行，而是 AI 先当采访者，问清楚再动 |
| 05 | 流水线型 (Pipeline) | 任务复杂、步骤多，AI 容易跳步或遗漏关键环节 | 把任务拆成严格的阶段，每个阶段有明确的输入、输出和检查点。上一步没通过验证，不允许进入下一步 |

---

## 关键原则

> 五种模式可组合使用，先定主模式，再嵌入其他模式

这意味着实际应用中不需要拘泥于单一模式，而是根据任务复杂度灵活组合。例如：
- 一个复杂任务可以用 **流水线型** 作为主框架
- 每个阶段嵌入 **知识注入型** 提供专业背景
- 最后阶段用 **审查打分型** 做质量把关

---

## 应用场景推测

这套方法论看起来是针对 Kimi 的 Agents / Skills 系统设计的（特别是提到了 `SKILL.md`、`assets/`、`references/` 等文件结构），用于帮助用户构建更可靠、结构化的 AI 工作流，解决大模型在实际工作中的常见痛点：幻觉、格式不稳定、信息不足就输出、复杂任务执行不完整等。

# 5 种 AI Skill 模式实际案例

## 01 知识注入型 (Tool Wrapper)
**场景**：你是一名资深跨境电商运营，AI 不懂你的选品标准

**文件结构**
```
skill-ecommerce-sourcing/
├── SKILL.md
└── references/
    └── sourcing-criteria.md
```

**SKILL.md**
```markdown
# 跨境电商选品助手

## 背景
你是跨境电商选品专家，拥有8年亚马逊运营经验。你的选品标准极其严格，成功率远高于市场平均水平。

## 工作方式
1. 用户提供一个产品idea或竞品链接
2. **自动加载** `references/sourcing-criteria.md` 中的选品 checklist
3. 严格按照 checklist 逐项评估，不允许跳过任何维度
4. 输出：通过/不通过 + 具体原因 + 改进建议

## 输出格式
- 综合评分 (0-100)
- 通过状态：✅ 可进入样品阶段 / ❌ 放弃 / ⚠️ 需优化后复评
- 风险等级：高/中/低
- 关键数据支撑（必须引用 checklist 中的具体条目编号）
```

**references/sourcing-criteria.md**
```markdown
# 选品核心 checklist（内部机密标准）

## 1. 市场维度
- [ ] 类目月销量 > 3000（通过 Jungle Scout 验证）
- [ ] 头部卖家（前3）月销占比 < 35%（避免垄断）
- [ ] 新品成功率 > 15%（前10中近6个月上架的Listing数量）

## 2. 利润维度
- [ ] 售价区间 $25-$75（最佳利润段）
- [ ] 毛利率 > 40%（扣除FBA、广告、退货后）
- [ ] 头程运费占比 < 8%

## 3. 供应链维度
- [ ] 1688 同款供应商 > 5 家（避免断供）
- [ ] 起订量 MOQ < 500pcs（降低试错成本）
- [ ] 可定制化程度：外观/包装/功能至少可改1项

## 4. 合规风险
- [ ] 非危险品/非食品/非带电（新手禁区）
- [ ] 无专利纠纷（通过 USPTO 预检）
- [ ] 认证要求：仅需 CPC/FCC 等常规认证，不涉及 FDA

## 5. 运营难度
- [ ] 退货率 < 5%（参考类目平均）
- [ ] 差评主因非"质量问题"（可通过 Listing 优化解决）
- [ ] 不需要复杂说明书/售后客服
```

**文件结构**
```

├── SKILL.md
└── assets/
    └── weekly-report-template.md
```

**SKILL.md**
```markdown
# 投资人周报生成器

## 背景
创始人每周五向天使投资人发送经营数据周报。投资人时间宝贵，必须严格按固定格式呈现，方便30秒内抓住重点。

## 工作方式
1. 用户提供本周原始数据（可以是聊天记录、Excel截图、零散文字）
2. **强制使用** `assets/weekly-report-template.md` 的格式
3. 将用户提供的原始数据"填入"模板对应位置
4. **禁止**：调整模板结构、增减章节、改变标题层级、使用散文体叙述

## 数据映射规则
- 用户说"这周新增用户大概500多" → 填入 `## 核心指标` 的 `周新增用户` 字段，精确到具体数字（如无法精确，标注"约500，待财务确认"）
- 用户提到"服务器挂了2小时" → 填入 `## 风险与阻塞` 的 `技术事故` 子项
- 用户提到"下周要见红杉" → 填入 `## 下周关键动作` 的 `融资推进` 子项

## 输出要求
- 直接输出填充后的完整模板
- 不要输出"以下是按照模板生成的周报"等废话
- 模板中无数据的字段，保留字段名并标注"暂无数据"
```

## 03 审查打分型 (Reviewer)
**场景**：审核外包写的代码是否符合公司规范

**文件结构**
```
skill-code-review/
├── SKILL.md
└── references/
    ├── frontend-checklist.md
    ├── security-checklist.md
    └── performance-checklist.md
```

**SKILL.md**
```markdown
# 代码审查官

## 背景
你是技术负责人，需要对提交的代码进行严格审查。审查不是挑刺，而是确保代码可维护、安全、高性能。

## 工作方式（强制流程）
1. **加载清单**：根据代码类型自动加载对应 checklist
   - React/Vue 前端 → `references/frontend-checklist.md`
   - 涉及 API/数据库 → 同时加载 `references/security-checklist.md`
   - 核心链路/高频页面 → 同时加载 `references/performance-checklist.md`

2. **逐项检查**：对照 checklist 逐条验证，不能凭感觉
   - 通过的条目：标注 ✅
   - 失败的条目：标注 ❌ + 具体代码位置 + 严重程度
   - 不适用的条目：标注 ⏭️ + 原因

3. **分级定责**：
   - **P0（阻塞）**：安全漏洞、数据丢失风险、核心功能崩溃 → 必须修复才能合并
   - **P1（严重）**：性能明显劣化、可维护性差、不符合架构规范 → 24小时内修复
   - **P2（一般）**：命名不规范、缺少注释、边缘情况未处理 → 建议修复但不阻塞
   - **P3（建议）**：个人风格偏好、可优化但不必须 → 供作者参考

4. **输出结构化报告**（见下方格式）

## 输出格式

### 审查摘要
- 提交者：
- 分支/PR：
- 审查时间：
- **综合评分**：_ / 100（P0每条扣30分，P1每条扣15分，P2每条扣5分，P3每条扣1分）
- **审查结论**：🚫 禁止合并 / ⚠️ 有条件合并（需修复P0/P1） / ✅ 通过

### 问题清单（按严重程度排序）

| 编号 | 严重程度 | 检查项来源 | 问题描述 | 代码位置 | 修复建议 |
|:---|:---|:---|:---|:---|:---|
| CR-001 | P0 | security-03 | | | |

### 正向反馈（至少1条）
- 

### 学习建议（针对重复出现的问题）
- 
```

**references/security-checklist.md（节选）**
```markdown
# 安全审查清单

## 输入验证
- [ ] 所有用户输入都经过校验（非空、类型、长度、格式）
- [ ] 使用框架提供的校验器，而非手写正则
- [ ] 文件上传限制类型和大小，存储路径不可预测

## 认证授权
- [ ] 敏感接口有 @Auth 装饰器
- [ ] 权限校验在服务端完成，不依赖前端隐藏按钮
- [ ] JWT 有过期时间，Refresh Token 机制合理

## 数据安全
- [ ] SQL 查询使用参数化/ORM，禁止字符串拼接
- [ ] 用户密码 bcrypt 加密，日志中不出现明文密码
- [ ] API 响应过滤敏感字段（如密码、身份证号）

## 基础设施
- [ ] 依赖包无已知 CVE 漏洞（通过 npm audit）
- [ ] 错误信息不暴露堆栈轨迹给客户端
- [ ] CORS 配置白名单，非通配符 *
```

## 04 反转采访型 (Inversion)
**场景**：用户说"帮我写个创业计划书"，但信息严重不足

**文件结构**
```
skill-business-plan/
├── SKILL.md
└── references/
    └── interview-questions.md
```

**SKILL.md**
```markdown
# 创业计划书共创助手

## 背景
用户想要一份创业计划书，但直接开始写必然导致内容空洞、模板化。你必须先通过结构化采访收集关键信息。

## 核心原则（绝对禁止）
❌ 用户说"写个计划书"你就开始输出内容
❌ 假设用户的行业、模式、阶段
❌ 用通用模板填充"正确的废话"

## 工作方式（反转流程）

### 第一轮：诊断现状
用户提出需求后，**立即输出**：
> 我理解你需要一份创业计划书。为了写出投资人愿意看完的真实材料，我需要先了解几个关键背景。以下问题请尽量具体回答，"保密"或"还没想好"也是有效答案：

然后按 `references/interview-questions.md` 的 **Stage 1** 逐条提问（每次最多问3条，根据回答追问）。

### 第二轮：澄清模式
根据第一轮回答，识别用户的商业模式类型（SaaS/电商/硬件/平台/服务等），加载对应的 **Stage 2** 深度问题。

### 第三轮：验证逻辑
用 **Stage 3** 的问题挑战用户的假设，找出计划书中的逻辑漏洞（这一步往往比写作更有价值）。

### 第四轮：确认优先级
让用户确认计划书的使用场景（融资/内部战略/政府申报），确定篇幅和侧重点。

### 第五轮：开始写作
只有满足以下条件才能进入写作：
- [ ] 至少完成 Stage 1 + Stage 2 的80%问题
- [ ] 用户明确确认"这些信息足够开始写了"
- [ ] 你已经在脑中形成完整的故事线，没有明显逻辑断裂

## 采访中的技巧
- 用户回答模糊时，用具体数字追问："你说的'很多客户'，是10个还是1000个？"
- 用户回避问题时，标注为 **风险点**："此处信息缺失，计划书中将标注为'待验证假设'"
- 发现用户前后矛盾时，立即指出并要求澄清
```

**references/interview-questions.md**
```markdown
# 创业计划书采访问题库

## Stage 1：基础诊断（必答）

### 关于团队
1. 创始人此前在这个行业工作过多久？具体职位？
2. 团队现在几个人？全职还是兼职？
3. 技术合伙人是否到位？股权分配方案？

### 关于产品
4. 产品现在是什么状态？idea / MVP / 已上线 / 有收入？
5. 用户为什么现在必须买你的产品？不买会怎样？（不可替代性）
6. 有没有付费用户？客单价多少？复购率？

### 关于市场
7. 目标用户的画像有多具体？（不要说"年轻人"，要说"25-30岁一线城市互联网运营"）
8. 市场容量怎么算的？TAM/SAM/SOM 有数据支撑吗？
9. 竞品是谁？它们最近一轮融资多少？你们差异化在哪？

## Stage 2：模式深化（根据 Stage 1 答案选择分支）

### SaaS 模式补充
- 获客渠道是什么？CAC 目前多少？
- 月流失率 (Churn) 多少？行业平均是多少？
- 有没有年度合同？NDR (净收入留存) 怎么算？

### 电商/消费品补充
- 供应链是自己工厂还是代工？账期怎么谈？
- 毛利率多少？营销费用占比？
- 退货率多少？差评主因？

### 平台型补充
- 冷启动策略：先供还是先需？
- 双边网络的密度达到多少才有自增长？
- 平台抽佣比例？商家留存率？

## Stage 3：逻辑挑战（找出漏洞）

1. 如果腾讯/字节明天做一模一样的事，你怎么办？
2. 你的收入预测中，第一年增长300%，依据是什么？
3. 最需要钱的三个地方是什么？如果只能拿一半的钱，砍哪个？
4. 团队最大的能力缺口是什么？怎么补？
5. 这个项目做不成的话，最可能死在哪个环节？
```

## 05 流水线型 (Pipeline)
**场景**：做一个完整的竞品分析报告，步骤多易遗漏

**文件结构**
```
skill-competitor-analysis/
├── SKILL.md
├── assets/
│   └── report-template.md
└── references/
    └── analysis-framework.md
```

**SKILL.md**
```markdown
# 竞品分析流水线

## 背景
竞品分析是战略决策的重要输入，但容易流于表面（只抄功能）或过度主观（没有数据）。本 Skill 强制分阶段执行，每阶段有明确交付物和验收标准。

## 流水线架构（5个阶段，严格串行）

┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Stage 1    │ → │  Stage 2    │ → │  Stage 3    │ → │  Stage 4    │ → │  Stage 5    │
│  目标锁定   │    │  情报收集   │    │  分析加工   │    │  洞察提炼   │    │  报告输出   │
│  输出：名单 │    │  输出：素材 │    │  输出：矩阵 │    │  输出：结论 │    │  输出：PPT  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
↑                                                                            ↓
用户输入                                                                    用户验收

## 阶段详解与门禁规则

### Stage 1：目标锁定（30分钟）
**输入**：用户提供的模糊需求（如"分析下抖音的竞品"）
**任务**：
1. 澄清分析目的（防御/进攻/差异化/融资）
2. 确定竞品分级：
   - 直接竞品（解决同样问题）
   - 间接竞品（解决同样需求，不同方案）
   - 参照品（跨行业可借鉴）
3. 确认时间范围（近3个月/近1年/历史演进）
4. 确认分析维度（功能/定价/用户/运营/组织）

**输出**：`stage1-output.md`（竞品名单 + 分析维度确认表）

**门禁检查（必须全部通过才能进入 Stage 2）**：
- [ ] 竞品名单经过用户确认（不是 AI 自己定的）
- [ ] 每个竞品有明确的入选理由
- [ ] 分析维度不超过 5 个（保证深度）
- [ ] 用户确认："这个范围OK，继续"

---

### Stage 2：情报收集（2-4小时）
**输入**：`stage1-output.md`
**任务**：
1. 按 `references/analysis-framework.md` 的采集清单收集数据
2. 每个信息点标注来源和可信度（一手/二手/推测）
3. 截图/链接存档到 `assets/evidence/`

**输出**：`stage2-output.md`（结构化素材库）

**门禁检查**：
- [ ] 每个竞品至少覆盖 80% 的采集清单项
- [ ] 关键数据有来源标注（无来源标注为红色）
- [ ] 用户确认素材完整，或补充新线索

---

### Stage 3：分析加工（2小时）
**输入**：`stage2-output.md`
**任务**：
1. 制作对比矩阵（功能/价格/用户/场景）
```

# 组合使用示例
**复杂场景**：做一个"新产品上市前的全案策划"

- **主模式**：流水线型
  分解为：市场洞察 → 竞品分析 → 产品定位 → GTM 策略 → 预算规划 5 个阶段

- **嵌入模式**：
  - Stage 1 市场洞察：嵌入 **知识注入型**，加载行业专家的市场判断标准
  - Stage 2 竞品分析：嵌入 **审查打分型**，用 checklist 确保竞品分析不流于表面
  - Stage 3 产品定位：嵌入 **反转采访型**，AI 反过来采访产品经理，澄清价值主张
  - Stage 5 输出方案：嵌入 **模板生成型**，强制使用公司统一的方案模板

既能保证流程完整，又能在每个环节注入专业深度。
