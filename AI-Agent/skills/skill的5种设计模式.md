## 一、五种核心 Skill 设计模式

| 模式名称 | 核心解决问题 | 核心思路与实现 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **知识注入型 (Tool Wrapper)** | AI 通用知识不精准，需要注入领域/私有知识 | 1. 在 `SKILL.md` 中定义触发条件<br>2. 知识文件（如 `conventions.md`）存放在 `references/` 目录<br>3. 触发时按需加载，不污染无关上下文 | 前端 React 规范、数据库 SQL 规范、企业内部开发标准等 |
| **模板生成型 (Generator)** | AI 输出格式不统一、结构飘忽不定 | 1. `assets/` 目录存放输出模板（如 `report-template.md`）<br>2. `references/` 存放风格指南（如语气、命名规则）<br>3. 强制 AI 按模板结构填充内容，保证一致性 | API 文档生成、测试报告、周报/日报、技术方案模板等 |
| **审查打分型 (Reviewer)** | 需要 AI 按固定标准检查/评估内容 | 1. `SKILL.md` 定义审查流程：加载清单→逐项检查→按严重程度分类→生成报告<br>2. `references/` 存放检查清单（如 `review-checklist.md`）<br>3. 流程与规则解耦，换清单即可切换审查标准 | 代码规范审查、安全漏洞检查、文档合规性校验、设计评审等 |
| **反转采访型 (Inversion)** | AI 信息不足时盲目执行，结果偏差大 | 1. 流程反转：先提问，再执行<br>2. 分三阶段：<br>① 发现阶段：问清问题背景、用户、规模<br>② 约束阶段：确认平台、技术栈、需求边界<br>③ 合成阶段：信息齐全后再生成结果 | 客服机器人方案设计、产品需求拆解、系统架构规划等 |
| **流水线型 (Pipeline)** | 复杂任务 AI 易跳步/遗漏关键环节 | 1. 任务拆分为严格的阶段，每个阶段定义明确的输入、输出和检查点<br>2. 阶段间设置"关卡（Gate）"，上一步未通过验证无法进入下一步 | API 文档自动生成、代码重构、数据处理流程、多步骤自动化测试等 |

> 💡 关键提示：五种模式可**组合使用**，建议先确定主模式，再嵌入其他模式（比如流水线的某个步骤，可嵌入模板生成型输出）。

## 二、Skill 模式选型决策指南

用这个流程图快速判断该用哪种模式：

1. **先判断：Skill 是否要生成输出？**
   - ✅ 是 → 再判断：是否需要固定模板？
     - 是 → 用 **模板生成型（Generator）**
     - 否 → 用 **知识注入型（Tool Wrapper）**
   - ❌ 否 → 再判断：是否要评估已有输入？
     - 是 → 用 **审查打分型（Reviewer）**
     - 否 → 再判断：是否需要用户先输入信息？
       - 是 → 用 **反转采访型（Inversion）**
       - 否 → 再判断：是否有固定顺序的步骤？
         - 是 → 用 **流水线型（Pipeline）**
         - 否 → 用 **知识注入型（Tool Wrapper）**

## 三、5 种 Skill 设计模板 + 案例

以下是可直接复制使用的 5 种 Skill 设计模板 + 案例对照，按"先套模板，再填案例"的方式整理：

### 01 知识注入型 (Tool Wrapper)

#### 通用模板

```markdown
# [Skill 名称]

## 背景
你是 [领域] 的 [身份]，拥有 [X] 年经验。[一句话说明为什么通用 AI 做不好这件事]。

## 工作方式
1. 用户提供 [输入内容]
2. **自动加载** `references/[知识文件].md`
3. 严格按照 [知识文件] 中的 [checklist/标准/规则] 执行
4. 输出：[具体交付物]

## 输出格式
- [格式要求 1]
- [格式要求 2]
- 必须引用 [知识文件] 中的 [条目编号/规则名称]

## 禁止事项
- [禁止自由发挥的点 1]
- [禁止跳过/忽略的点 2]
```

#### 案例：跨境电商选品助手

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[领域]` | 跨境电商 |
| `[身份]` | 选品专家 |
| `[X] 年经验` | 8 年亚马逊运营 |
| `[输入内容]` | 一个产品 idea 或竞品链接 |
| `[知识文件]` | `sourcing-criteria.md`（含市场/利润/供应链/合规/运营 5 大维度 checklist） |
| `[交付物]` | 通过/不通过判定 + 综合评分 + 风险等级 + 改进建议 |
| `[禁止事项]` | 不允许跳过任何评估维度；必须引用 checklist 条目编号 |

### 02 模板生成型 (Generator)

#### 通用模板

```markdown
# [Skill 名称]

## 背景
[使用场景说明，强调"格式一致性"的重要性]。

## 工作方式
1. 用户提供 [原始数据/内容]
2. **强制使用** `assets/[模板文件].[扩展名]` 的格式
3. 将用户提供的内容"填入"模板对应位置
4. **禁止**：调整模板结构、增减章节、改变标题层级、自由发挥叙述

## 数据映射规则
- 用户说 "[原始表述示例]" → 填入 `[模板字段名]`
- 用户提到 "[原始表述示例]" → 填入 `[模板字段名]`
- [无数据时的处理方式]

## 输出要求
- 直接输出填充后的完整模板
- 不要输出"以下是按照模板生成的..."等废话
- 模板中无数据的字段，保留字段名并标注"[占位提示]"
```

#### 案例：投资人周报生成器

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[使用场景]` | 创始人每周五向天使投资人发送经营数据周报，投资人时间宝贵，必须 30 秒内抓住重点 |
| `[模板文件]` | `weekly-report-template.md`（含核心指标表/关键进展/风险阻塞/下周动作/资金状况 5 大固定章节） |
| `[原始表述→字段映射]` | "这周新增用户大概 500 多" → `周新增用户` 字段；"服务器挂了 2 小时" → `技术事故` 子项 |
| `[无数据处理]` | 保留字段名并标注"暂无数据" |

### 03 审查打分型 (Reviewer)

#### 通用模板

```markdown
# [Skill 名称]

## 背景
你是 [身份]，需要对 [审查对象] 进行 [审查目的]。

## 工作方式（强制流程）
1. **加载清单**：根据 [分类条件] 自动加载对应 checklist
   - [条件 1] → `[checklist 文件 1]`
   - [条件 2] → `[checklist 文件 2]`

2. **逐项检查**：对照 checklist 逐条验证
   - ✅ 通过
   - ❌ 失败 + 具体位置 + 严重程度
   - ⏭️ 不适用 + 原因

3. **分级定责**：
   - **P0（阻塞）**：[定义] → [处理方式]
   - **P1（严重）**：[定义] → [处理方式]
   - **P2（一般）**：[定义] → [处理方式]
   - **P3（建议）**：[定义] → [处理方式]

4. **输出结构化报告**

## 输出格式
### 审查摘要
- 对象：
- 时间：
- **综合评分**：_ / 100（[扣分规则]）
- **审查结论**：[结论选项]

### 问题清单（按严重程度排序）
| 编号 | 严重程度 | 检查项来源 | 问题描述 | 位置 | 修复建议 |

### 正向反馈（至少 [N] 条）
```

#### 案例：代码审查官

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[审查对象]` | 提交的代码 |
| `[分类条件]` | React/Vue 前端 → `frontend-checklist.md`；涉及 API/数据库 → 同时加载 `security-checklist.md` |
| `[P0 定义]` | 安全漏洞、数据丢失风险、核心功能崩溃 |
| `[P1 定义]` | 性能明显劣化、可维护性差、不符合架构规范 |
| `[扣分规则]` | P0 每条扣 30 分，P1 扣 15 分，P2 扣 5 分，P3 扣 1 分 |
| `[结论选项]` | 🚫 禁止合并 / ⚠️ 有条件合并 / ✅ 通过 |

### 04 反转采访型 (Inversion)

#### 通用模板

```markdown
# [Skill 名称]

## 背景
用户想要 [最终交付物]，但直接开始必然导致 [常见问题]。

## 核心原则（绝对禁止）
❌ 用户说"[触发词]"你就开始输出内容
❌ 假设用户的 [关键信息 1] / [关键信息 2]
❌ [其他常见错误]

## 工作方式（反转流程）

### 第一轮：[阶段名称]
用户提出需求后，**立即输出**：
> [引导语，说明为什么要先采访]

然后按 `references/[采访问题文件].md` 的 **Stage 1** 逐条提问（每次最多 [N] 条，根据回答追问）。

### 第二轮：[阶段名称]
根据第一轮回答，识别 [分类维度]，加载对应的 **Stage 2** 深度问题。

### 第三轮：[阶段名称]
用 **Stage 3** 的问题 [核心动作：挑战/验证/澄清]。

### 第四轮：[阶段名称]
[收尾确认动作]。

### 第五轮：开始 [最终动作]
只有满足以下条件才能进入：
- [ ] [条件 1]
- [ ] [条件 2]
- [ ] [条件 3]

## 采访技巧
- 用户回答模糊时：[追问方式]
- 用户回避问题时：[处理方式]
- 发现矛盾时：[处理方式]
```

#### 案例：创业计划书共创助手

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[最终交付物]` | 创业计划书 |
| `[常见问题]` | 内容空洞、模板化 |
| `[触发词]` | "帮我写个创业计划书" |
| `[采访问题文件]` | `interview-questions.md`（含 Stage 1 基础诊断 / Stage 2 模式深化 / Stage 3 逻辑挑战） |
| `[引导语]` | "为了写出投资人愿意看完的真实材料，我需要先了解几个关键背景" |
| `[分类维度]` | 商业模式类型（SaaS/电商/硬件/平台/服务） |
| `[核心动作]` | 挑战用户的假设，找出计划书中的逻辑漏洞 |
| `[进入条件]` | 完成 Stage 1+2 的 80% 问题；用户明确确认；脑中形成完整故事线 |

### 05 流水线型 (Pipeline)

#### 通用模板

```markdown
# [Skill 名称]

## 背景
[任务名称] 是 [重要性说明]，但容易 [常见失败模式]。本 Skill 强制分阶段执行。

## 流水线架构（[N] 个阶段，严格串行）

[用 ASCII 或文字描述阶段流转图]

## 阶段详解与门禁规则

### Stage 1：[阶段名]（[预估时间]）
**输入**：[上一阶段输出 / 用户初始输入]
**任务**：
1. [具体动作 1]
2. [具体动作 2]
3. [具体动作 3]

**输出**：`stage1-output.[格式]`

**门禁检查（必须全部通过才能进入 Stage 2）**：
- [ ] [检查项 1]
- [ ] [检查项 2]
- [ ] [检查项 3]

---

### Stage 2：[阶段名]（[预估时间]）
**输入**：`stage1-output.[格式]`
**任务**：
1. [具体动作 1]
2. [具体动作 2]

**输出**：`stage2-output.[格式]`

**门禁检查**：
- [ ] [检查项 1]
- [ ] [检查项 2]

[继续 Stage 3...N]

## 紧急通道
如果用户说"[紧急触发词]"：
- [简化规则 1]
- [简化规则 2]
- **必须**在最终输出中标注："本分析基于简化流程，置信度为 [X]"
```

#### 案例：竞品分析流水线

| 模板占位符 | 实际填充 |
| :--- | :--- |
| `[任务名称]` | 竞品分析 |
| `[常见失败模式]` | 流于表面（只抄功能）或过度主观（没有数据） |
| `[N] 个阶段` | 5 个阶段：目标锁定 → 情报收集 → 分析加工 → 洞察提炼 → 报告输出 |
| `Stage 1 任务` | 澄清分析目的、确定竞品分级（直接/间接/参照）、确认时间范围、确认分析维度 |
| `Stage 1 门禁` | 竞品名单经用户确认；每个竞品有入选理由；分析维度≤5 个；用户说"继续" |
| `Stage 2 任务` | 按 `analysis-framework.md` 采集清单收集数据，标注来源和可信度 |
| `Stage 2 门禁` | 每个竞品覆盖 80% 采集项；关键数据有来源；用户确认素材完整 |
| `Stage 3 任务` | 制作对比矩阵、绘制用户旅程对比图、识别战略假设、找出市场空白点 |
| `Stage 3 门禁` | 矩阵有证据支撑；至少发现 1 个验证过的空白点；用户确认逻辑成立 |
| `Stage 4 任务` | 回答"对我们意味着什么"、提炼 3 个可执行建议、区分立即做/观察/放弃 |
| `Stage 4 门禁` | 建议与 Stage 1 目的相关；有明确负责人和时间；用户确认有价值 |
| `Stage 5 任务` | 按模板生成报告、Executive Summary 1 页、附录保留全部交付物 |
| `Stage 5 门禁` | 用户验收通过；保留 Stage 1-4 交付物；含"局限性声明" |
| `[紧急触发词]` | "明天就要，简化流程" |
| `[简化规则]` | 跳过部分采集项；矩阵缩减为 2 个维度 |

## 四、完整 SKILL.md 实战案例

我来为你设计每种模式的实际 SKILL.md 案例，基于 Kimi Agents 的文件结构规范（`SKILL.md` + `assets/` + `references/`）。

### 01 知识注入型：跨境电商选品助手

**场景**：你是一名资深跨境电商运营，AI 不懂你的选品标准

**文件结构**：

```
skill-ecommerce-sourcing/
├── SKILL.md
└── references/
    └── sourcing-criteria.md
```

**SKILL.md**：

```markdown
# 跨境电商选品助手

## 背景
你是跨境电商选品专家，拥有 8 年亚马逊运营经验。你的选品标准极其严格，成功率远高于市场平均水平。

## 工作方式
1. 用户提供一个产品 idea 或竞品链接
2. **自动加载** `references/sourcing-criteria.md` 中的选品 checklist
3. 严格按照 checklist 逐项评估，不允许跳过任何维度
4. 输出：通过/不通过 + 具体原因 + 改进建议

## 输出格式
- 综合评分 (0-100)
- 通过状态：✅ 可进入样品阶段 / ❌ 放弃 / ⚠️ 需优化后复评
- 风险等级：高/中/低
- 关键数据支撑（必须引用 checklist 中的具体条目编号）
```

**references/sourcing-criteria.md**：

```markdown
# 选品核心 checklist（内部机密标准）

## 1. 市场维度
- [ ] 类目月销量 > 3000（通过 Jungle Scout 验证）
- [ ] 头部卖家（前 3）月销占比 < 35%（避免垄断）
- [ ] 新品成功率 > 15%（前 10 中近 6 个月上架的 Listing 数量）

## 2. 利润维度
- [ ] 售价区间 $25-$75（最佳利润段）
- [ ] 毛利率 > 40%（扣除 FBA、广告、退货后）
- [ ] 头程运费占比 < 8%

## 3. 供应链维度
- [ ] 1688 同款供应商 > 5 家（避免断供）
- [ ] 起订量 MOQ < 500pcs（降低试错成本）
- [ ] 可定制化程度：外观/包装/功能至少可改 1 项

## 4. 合规风险
- [ ] 非危险品/非食品/非带电（新手禁区）
- [ ] 无专利纠纷（通过 USPTO 预检）
- [ ] 认证要求：仅需 CPC/FCC 等常规认证，不涉及 FDA

## 5. 运营难度
- [ ] 退货率 < 5%（参考类目平均）
- [ ] 差评主因非"质量问题"（可通过 Listing 优化解决）
- [ ] 不需要复杂说明书/售后客服
```

### 02 模板生成型：投资人周报生成器

**场景**：每周要给投资人发数据周报，AI 每次格式都不一样

**文件结构**：

```
skill-investor-weekly/
├── SKILL.md
└── assets/
    └── weekly-report-template.md
```

**SKILL.md**：

```markdown
# 投资人周报生成器

## 背景
创始人每周五向天使投资人发送经营数据周报。投资人时间宝贵，必须严格按固定格式呈现，方便 30 秒内抓住重点。

## 工作方式
1. 用户提供本周原始数据（可以是聊天记录、Excel 截图、零散文字）
2. **强制使用** `assets/weekly-report-template.md` 的格式

用户想要一份创业计划书，但直接开始写必然导致内容空洞、模板化。你必须先通过结构化采访收集关键信息。

❌ 假设用户的行业、模式、阶段
❌ 用通用模板填充"正确的废话"

## 工作方式（反转流程）

### 第一轮：诊断现状
用户提出需求后，**立即输出**：
> 我理解你需要一份创业计划书。为了写出投资人愿意看完的真实材料，我需要先了解几个关键背景。以下问题请尽量具体回答，"保密"或"还没想好"也是有效答案：

### 第四轮：确认优先级
让用户确认计划书的使用场景（融资/内部战略/政府申报），确定篇幅和侧重点。

### 第五轮：开始写作
只有满足以下条件才能进入写作：
- [ ] 至少完成 Stage 1 + Stage 2 的 80% 问题
- [ ] 用户明确确认"这些信息足够开始写了"
- [ ] 你已经在脑中形成完整的故事线，没有明显逻辑断裂

## 采访中的技巧
- 用户回答模糊时，用具体数字追问："你说的'很多客户'，是 10 个还是 1000 个？"

- 发现用户前后矛盾时，立即指出并要求澄清
```

# 竞品分析流水线

## 背景
竞品分析是战略决策的重要输入，但容易流于表面（只抄功能）或过度主观（没有数据）。本 Skill 强制分阶段执行，每阶段有明确交付物和验收标准。

## 流水线架构（5 个阶段，严格串行）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Stage 1    │ → │  Stage 2    │ → │  Stage 3    │ → │  Stage 4    │ → │  Stage 5    │
│  目标锁定   │    │  情报收集   │    │  分析加工   │    │  洞察提炼   │    │  报告输出   │
│  输出：名单 │    │  输出：素材 │    │  输出：矩阵 │    │  输出：结论 │    │  输出：PPT  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
↑                                                                            ↓
用户输入                                                                    用户验收
```

## 阶段详解与门禁规则

### Stage 1：目标锁定（30 分钟）
**输入**：用户提供的模糊需求（如"分析下抖音的竞品"）
**任务**：
1. 澄清分析目的（防御/进攻/差异化/融资）
2. 确定竞品分级：
   - 直接竞品（解决同样问题）
   - 间接竞品（解决同样需求，不同方案）
   - 参照品（跨行业可借鉴）
3. 确认时间范围（近 3 个月/近 1 年/历史演进）
4. 确认分析维度（功能/定价/用户/运营/组织）

**输出**：`stage1-output.md`（竞品名单 + 分析维度确认表）

**门禁检查（必须全部通过才能进入 Stage 2）**：
- [ ] 竞品名单经过用户确认（不是 AI 自己定的）
- [ ] 每个竞品有明确的入选理由
- [ ] 分析维度不超过 5 个（保证深度）
- [ ] 用户确认："这个范围 OK，继续"

---

### Stage 2：情报收集（2-4 小时）
**输入**：`stage1-output.md`
**任务**：
1. 按 `references/analysis-framework.md` 的采集清单收集数据
2. 每个信息点标注来源和可信度（一手/二手/推测）
3. 截图/链接存档到 `assets/evidence/`

**输出**：`stage2-output.md`（结构化素材库）

**门禁检查**：
- [ ] 每个竞品至少覆盖 80% 的采集清单项
- [ ] 关键数据有来源标注（无来源标注为红色）
- [ ] 用户确认素材完整，或补充新线索

---

### Stage 3：分析加工（2 小时）
**输入**：`stage2-output.md`
**任务**：
1. 制作对比矩阵（功能/价格/用户/场景）
2. 绘制用户旅程对比图
3. 识别各竞品的战略假设（它们认为什么最重要）
4. 找出市场空白点（用户抱怨多但无人做好的）

**输出**：`stage3-output.md`（分析矩阵 + 空白点地图）

**门禁检查**：
- [ ] 矩阵中每个单元格有具体证据支撑，非主观臆断
- [ ] 至少发现 1 个被验证的市场空白点
- [ ] 用户确认分析逻辑成立

---

### Stage 4：洞察提炼（1 小时）
**输入**：`stage3-output.md`
**任务**：
1. 回答核心问题："对我们意味着什么？"
2. 提炼 3 个可执行的战略建议
3. 每个建议有明确的假设、风险和验证方法
4. 区分：立即做 / 观察 / 放弃

**输出**：`stage4-output.md`（战略建议 + 行动清单）

**门禁检查**：
- [ ] 建议与 Stage 1 的分析目的直接相关
- [ ] 每个建议有明确的负责人和时间（如适用）
- [ ] 用户确认："这些建议对我有价值"

---

### Stage 5：报告输出（1 小时）
**输入**：`stage4-output.md` + `assets/report-template.md`
**任务**：
1. 按模板生成最终报告
2. Executive Summary 控制在 1 页
3. 附录包含完整数据支撑

**输出**：`final-report.md`

**门禁检查**：
- [ ] 报告经过用户验收（用户说"可以用了"才算完成）
- [ ] 所有 Stage 1-4 的交付物作为附录保留
- [ ] 报告中有"局限性声明"（数据缺口、分析边界）

## 紧急通道
如果用户说"明天就要，简化流程"：
- 跳过 Stage 2 的部分采集项（需明确记录哪些被跳过）
- Stage 3 的矩阵缩减为最关键的 2 个维度

## 五、组合使用示例

**复杂场景**：做一个"新产品上市前的全案策划"

**主模式**：流水线型（分解为市场洞察 → 竞品分析 → 产品定位 → GTM 策略 → 预算规划 5 个阶段）

**嵌入模式**：
- Stage 1 市场洞察：嵌入 **知识注入型**，加载行业专家的市场判断标准
- Stage 2 竞品分析：嵌入 **审查打分型**，用 checklist 确保竞品分析不流于表面
- Stage 3 产品定位：嵌入 **反转采访型**，AI 反过来采访产品经理，澄清价值主张
- Stage 5 输出方案：嵌入 **模板生成型**，强制使用公司统一的方案模板

这样既能保证流程完整，又能在每个环节注入专业深度。

| 模板生成型 | 输出格式飘忽不定 | `assets/` 下的模板文件 | "强制使用"、"往里填内容"、"禁止自由发挥" |
| 审查打分型 | 需要结构化评估质量 | `references/` 下的 checklist | "逐项检查"、"分级定责"、"P0/P1/P2" |
| 反转采访型 | AI 信息不足就瞎做 | `references/` 下的采访问题库 | "先当采访者"、"问清楚再动"、"禁止马上执行" |
| 流水线型 | 复杂任务易跳步遗漏 | `SKILL.md` 内的阶段定义 | "严格串行"、"门禁检查"、"上一步没通过不进下一步" |

## 组合速查

遇到复杂任务时，按这个顺序决策：

1. 任务步骤多、易遗漏？ → 先用 **流水线型** 搭骨架
2. 某个阶段需要专业判断？ → 该阶段嵌入 **知识注入型**
3. 阶段产出需要质量保证？ → 嵌入 **审查打分型**
4. 阶段输入信息不足？ → 嵌入 **反转采访型**
5. 最终交付物格式要求高？ → 最后阶段嵌入 **模板生成型**

## 七、9 大最佳实践（三层分类）

### 1. 内容层（写给模型看的规则）

- ① **Description 只写触发时机，不写摘要**：让模型清楚"什么时候用这个 Skill"，而不是做功能介绍。
- ② **不写显而易见的内容**：避免重复教材级知识，重点补上下文、踩过的坑和特殊约束。
- ③ **构建 Gotchas 部分**：把踩过的坑、常见错误单独列出来，这是模型最容易忽略的细节。

### 2. 结构层（文件系统与流程设计）

- ④ **文件系统 + 渐进式披露**：把 Skill 做成文件夹，按触发条件按需加载文件，避免一次性注入过多信息。
- ⑤ **用 `config.json` 管理配置**：把可变参数、开关抽离到配置文件，不用改核心逻辑就能调整行为。
- ⑥ **避免过度约束 AI**：只约束目标和边界，不强制限制 AI 的实现路径，给模型留发挥空间。

### 3. 高级技术层（提升 Skill 能力的进阶技巧）

- ⑦ **内存与数据存储**：让 Skill 写日志、记录状态，实现会话级记忆，支持上下文关联。
- ⑧ **存储脚本并生成代码**：直接给 AI 可执行的代码片段，而不是纯文档描述，减少理解偏差。
- ⑨ **按需 Hooks**：设置会话级"护栏"，用完自动失效，避免规则在后续对话中持续生效。

## 八、核心设计原则总结

1. **按需加载，不污染上下文**：知识、模板、规则都只在需要时加载，避免无关信息干扰模型。
2. **流程与规则解耦**：比如审查流程固定，检查清单可替换；模板骨架固定，内容填充灵活。
3. **强约束目标，弱约束路径**：只明确"要什么、不能有什么"，不限制 AI 的思考方式。
4. **复杂任务必须拆阶段 + 设关卡**：多步骤任务一定要做阶段拆分，防止跳步和遗漏。
